# Providers

How defib drives an external agent CLI. This document owns the **Provider interface**, the
**adapter contracts**, and the **per-provider specifics**. It does not repeat the process
model (see [architecture.md](architecture.md)), the detection rule format (see
[detection.md](detection.md)), or the config keys (see [configuration.md](configuration.md)).

> **Verification required.** The exact flags below are the *intended* contract but MUST be
> validated against the installed provider version before the adapter is considered done. Each
> adapter milestone in [TODO.md](../TODO.md) has a "verify flags" task. Do not assume a flag
> exists — check `--help` and pin the tested version in the adapter's doc comment.

## The Provider interface

Defined in `internal/provider`. Adapters implement this and register themselves. Keep adapters
free of scheduling/state-machine logic — they only build commands, parse sessions, and provide
detection rules.

```go
package provider

// Provider drives one external agent CLI.
type Provider interface {
    // Name is the stable identifier used in config and the CLI (e.g. "claude").
    Name() string

    // Capabilities describes what this provider supports.
    Capabilities() Capabilities

    // BuildStart returns the argv (argv[0] is the executable) to begin a NEW session for
    // the task, plus any environment additions. If the provider supports client-supplied
    // session ids, the returned command MUST embed task.SessionRef (pre-generated by defib).
    BuildStart(task TaskSpec) (Command, error)

    // BuildResume returns the argv to CONTINUE an existing session identified by sessionRef.
    // Only called when Capabilities.Resume is true and a sessionRef is known.
    BuildResume(task TaskSpec, sessionRef string) (Command, error)

    // ExtractSessionRef inspects a finished attempt's output and returns the provider's
    // session id if it can be discovered from output. Return ("", false) if unknown or if
    // the ref was pre-generated (defib already knows it).
    ExtractSessionRef(out AttemptOutput) (string, bool)

    // DetectionRules returns the provider's built-in rules (see detection.md). The engine
    // merges these with user-configured rules.
    DetectionRules() []detect.Rule

    // CheckAvailability optionally reports whether the provider is usable again after a
    // QUOTA_EXHAUSTED outcome. Return Unsupported if the provider offers no cheap check.
    CheckAvailability(ctx context.Context, task TaskSpec) (Availability, error)
}

type Capabilities struct {
    Resume            bool // supports native session resume
    ClientSuppliedID  bool // caller may choose the session id up front
    Headless          bool // supports non-interactive run with capturable output
    Interactive       bool // supports PTY-attached interactive run
    StructuredOutput  bool // can emit machine-readable (e.g. JSON) output
}

type Command struct {
    Argv []string          // argv[0] is the executable (resolved via config binary path or PATH)
    Env  map[string]string // additional env vars merged over the daemon environment
}

type TaskSpec struct {
    Prompt     string   // initial instruction (may be empty if PassthroughArgs carries it)
    Passthrough []string // extra argv appended after adapter-controlled flags (after `--`)
    Cwd        string
    Mode       string   // "headless" | "interactive"
    SessionRef string   // pre-generated id if Capabilities.ClientSuppliedID, else ""
    Model      string   // optional provider model override from config
    ProviderConfig map[string]any // provider-scoped config block (see configuration.md)
}

type Availability struct {
    State   AvailabilityState // Available | Unavailable | Unsupported
    NextAt  *time.Time        // optional hint for when it will be available
}

// AttemptOutput carries a finished Attempt's exit code and captured output
// tails (bounded like detection input) for session extraction.
type AttemptOutput struct {
    ExitCode int
    Stdout   []byte
    Stderr   []byte
}
```

### Session strategy (important)

- **Preferred: pre-generate.** If `Capabilities.ClientSuppliedID` is true, defib generates a
  UUID at Task creation, stores it as `session_ref`, and the adapter passes it to the provider
  on the first Attempt. This means resume never depends on parsing output. Use this whenever
  the provider allows it.
- **Fallback: parse.** If the provider assigns its own id, `ExtractSessionRef` parses it from
  the first Attempt's output; the Daemon stores it before any Resume.
- **Existing session (`--session <id>`):** when the user points defib at an existing Session,
  the Task is created with `session_mode=existing` and `session_ref=<id>`; the very first
  Attempt uses `BuildResume`, not `BuildStart`.

### Mode selection

- Default mode is **headless** because output is easy to capture and classify, and it runs
  unattended. **Interactive** (PTY) mode is used only when the user asks for it (`--mode
  interactive`) and the provider advertises `Capabilities.Interactive`; the child runs on a
  pseudo-terminal the Daemon owns and a Client drives via `attach` (see
  [architecture.md](architecture.md#interactive-attach)).

## Claude Code adapter (`internal/provider/claude`) — first-class

**Executable:** resolved from `providers.claude.binary` (default `claude` on `PATH`).

**Intended capabilities:** `Resume=true`, `ClientSuppliedID=true`, `Headless=true`,
`Interactive=true`, `StructuredOutput=true`.

**Start (headless, new session):**
```
claude -p "<prompt>" \
  --session-id <defib-generated-uuid> \
  --output-format stream-json --verbose \
  [--model <model>] \
  [--permission-mode <mode> | <unattended approvals flag if opted in>] \
  [-- <passthrough args>]
```
- `--session-id` lets defib choose the id up front (pre-generate strategy).
- `--output-format stream-json` gives structured events that make detection and session
  extraction robust. Fall back to text if unavailable (verify).

**Resume (headless):**
```
claude -p "<continue-instruction-or-empty>" --resume <session-id> --output-format stream-json --verbose [--model …]
```
- If the continue instruction is empty, defib sends a minimal "continue" nudge configured in
  `providers.claude.resume_prompt` (default: `"Continue the previous task."`).

**Session extraction:** with `--session-id` supplied, `ExtractSessionRef` returns
`("", false)` (already known). If the id was not supplied, parse the `session_id` field from
the stream-json init event.

**Unattended approvals:** Claude Code will block on tool-permission prompts in headless mode
unless configured otherwise. defib must NOT enable the "skip permissions" flag implicitly. It
is enabled only when the user sets `providers.claude.unattended = true` (or passes the CLI
opt-in), which makes defib add the provider's skip-approvals flag and print the security
warning from [architecture.md](architecture.md#security-model). Verify the exact flag name for
the installed version.

**Detection:** see the Claude rule set in [detection.md](detection.md).

**Availability:** Claude typically exposes reset timing in the limit message itself, so
`CheckAvailability` returns `Unsupported` by default and defib relies on the extracted Reset
Time. A user may configure an `availability.command` probe instead.

## The fake provider (`internal/provider/fake`) — required for testing

A deterministic provider used by tests and local development (`--provider fake`). It reads a
script (from `--fake-script <file>` or the `providers.fake.script` config) describing what to
emit and how to exit, so the entire supervisor/scheduler/recovery loop can be exercised
**without touching a real provider or spending credits**.

Script format (one directive per line, executed in order for successive Attempts):
```
attempt: emit "line to stdout"       # write a line
attempt: emit-err "line to stderr"
attempt: sleep 200ms                 # simulate work
attempt: reset-at +2s                # emit a parseable reset-time hint
attempt: reply "ECHO: "              # read one line of PTY input, echo it back with a prefix
attempt: exit 1                      # end this Attempt with exit code
# blank line separates Attempt scripts; Attempt N uses the Nth block
```
The fake adapter ships built-in detection rules that recognize its own `reset-at` hint and a
`QUOTA_EXHAUSTED` marker line, enabling deterministic tests of every Outcome path. It supports
`ClientSuppliedID=true` and `Resume=true` (resume simply advances to the next Attempt block),
and `Interactive=true`: under `--mode interactive` the child runs on a PTY and the `reply`
directive blocks reading a line of forwarded input, so `attach` passthrough is testable without
a real provider.

## Adding a new provider — checklist

1. Create `internal/provider/<name>` implementing `Provider`.
2. Decide session strategy (pre-generate vs parse) based on the CLI's real capabilities.
3. Write `BuildStart`/`BuildResume` and verify flags against `--help` of a pinned version.
4. Add a default detection rule set in [detection.md](detection.md) and return it from
   `DetectionRules()`.
5. Register the provider in the registry init.
6. Add config keys under `providers.<name>` in [configuration.md](configuration.md).
7. Add table-driven tests using recorded real output fixtures in `testdata/`.
8. Update the provider table in [README.md](../README.md).
